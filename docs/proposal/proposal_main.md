## 1. 提案の概要

### 1.1 本提案の位置づけ

本提案は、デジタル庁が推進する「第三者検証業務におけるドキュメントレビュー支援AIエージェントの設計開発に係る調査研究」に対する技術提案書です。

弊社は、先行PoCの成果を踏まえ、**2ヶ月間の技術検証**により、生成AIを活用したドキュメントレビュー支援の**技術的実現可能性を明らかにする**ことを目的として本業務を遂行いたします。

### 1.2 提案のコンセプト

```mermaid
mindmap
  root((SmartReviewer))
    技術検証
      MCP準拠アーキテクチャ
      LLM/RAG統合
      庁内AI環境適合
    課題解決
      チェックリスト品質
      大容量文書処理
      是正提案の妥当性
    次年度への橋渡し
      作業計画試案
      技術資産の継承
      課題の明確化
```

### 1.3 期待される効果

| 効果 | 内容 |
|------|------|
| **技術的実現可能性の明確化** | 具体的な評価指標とともに実現可能性を検証 |
| **課題と対応方針の整理** | 技術面・運用面・セキュリティ面の課題を整理 |
| **年間計画試案の策定** | 令和8年度本格導入に向けた計画を策定 |

## 2. 業務理解

### 2.1 調達の背景

```mermaid
flowchart LR
    subgraph 現状課題
        A[レビュー業務の属人化]
        B[専門人材への過度な負荷]
        C[品質のばらつき]
    end
    
    subgraph 解決の方向性
        D[AI活用による効率化]
        E[品質の標準化]
        F[担当者負担の軽減]
    end
    
    A --> D
    B --> E
    C --> F
```

### 2.2 先行PoCで明らかになった課題

| # | 課題 | 概要 | 本提案での対応 |
|---|------|------|---------------|
| 1 | チェックリスト品質 | AI解釈可能な形式への変換 | 構造化チェックリスト設計 |
| 2 | 大容量文書処理 | 数万〜数十万文字の効率的処理 | 文書分割・階層処理方式 |
| 3 | 是正提案の妥当性 | 継続的な評価・改善基盤 | 回帰テスト環境構築 |
| 4 | 庁内AI環境への適合 | 源内の技術的制約への対応 | 制約考慮設計 |

### 2.3 MVPスコープ

本調査研究は2ヶ月間の技術検証であることから、以下の範囲に限定したMVPを開発します。

| 項目 | 範囲 |
|------|------|
| チェック対象文書種別 | 2種類（基本設計書、全体テスト計画書） |
| チェック項目 | 各文書上位10項目程度（計20項目程度） |
| 評価用データセット | 対象文書各10件程度（合計20件程度） |

## 4. 技術提案概要

### 4.1 システムアーキテクチャ

```mermaid
flowchart TB
    subgraph Host["MCP Host"]
        CLI[CLI Interface]
        UI[Web UI<br/>源内対応]
    end
    
    subgraph Servers["MCP Servers"]
        Core[smartreviewer-core<br/>文書管理・レビュー実行]
        RAG[smartreviewer-rag<br/>ベクトル検索・RAG]
        Knowledge[smartreviewer-knowledge<br/>チェックリスト・ナレッジ]
    end
    
    subgraph Data["Data Layer"]
        VDB[(Vector DB<br/>Qdrant)]
        KG[(Knowledge Graph)]
        Doc[(Document Store)]
    end
    
    subgraph LLM["LLM Layer"]
        Model[庁内利用可能LLM<br/>Claude/GPT-4等]
    end
    
    CLI --> Core
    UI --> Core
    Core <--> RAG
    Core <--> Knowledge
    RAG --> VDB
    Knowledge --> KG
    Core --> Doc
    Core <--> Model
    RAG <--> Model
```

### 4.2 先行PoC課題への対応方針

#### 課題1: チェックリスト品質

```mermaid
flowchart LR
    A[既存チェックリスト<br/>自然言語] --> B[構造化変換<br/>JSON/YAML]
    B --> C[AI解釈可能形式<br/>プロンプト埋込]
    C --> D[チェック実行<br/>LLM判定]
```

#### 課題2: 大容量文書処理

```mermaid
flowchart TB
    A[入力文書<br/>数万〜数十万文字] --> B{文書サイズ判定}
    B -->|小| C[直接処理]
    B -->|中| D[セクション分割処理]
    B -->|大| E[階層的要約処理]
    
    D --> F[関連セクション抽出]
    E --> G[軽量モデルで事前要約]
    G --> F
    
    C --> H[LLMチェック実行]
    F --> H
    H --> I[結果統合]
```

#### 課題3: 是正提案の妥当性評価

```mermaid
flowchart LR
    subgraph 評価基盤
        A[評価データセット] --> B[自動テスト実行]
        B --> C[精度計算]
        C --> D[レポート生成]
    end
    
    subgraph 改善サイクル
        D --> E[課題分析]
        E --> F[改善実装]
        F --> A
    end
```

### 4.3 KPI設定

| 指標 | 目標値 | 測定方法 |
|------|--------|---------|
| チェック合致率（Accuracy） | ≥70% | 評価データセットでの測定 |
| 適合率（Precision） | ≥70% | 指摘の正確性 |
| 再現率（Recall） | ≥70% | 指摘の網羅性 |
| 処理時間 | ≤30秒/文書 | 実測 |
| 再現性 | ≥95% | 複数回実行の一致率 |

## 8. リスク管理

### 8.1 リスク一覧

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|--------|--------|----------|------|
| 1 | LLM精度が目標未達 | 高 | 中 | 複数LLM比較、プロンプトチューニング |
| 2 | 大容量文書処理の性能問題 | 中 | 中 | 段階的処理、キャッシュ活用 |
| 3 | 庁内AI環境の制約判明 | 高 | 低 | 早期の要件確認、代替案の準備 |
| 4 | 評価データセット不足 | 中 | 低 | 合成データ生成、段階的拡充 |

### 8.2 リスク対応フロー

```mermaid
flowchart TB
    A[リスク検知] --> B{影響度評価}

    B -->|中低| D[週次報告で共有]
    
    C --> E[対策立案]
    D --> E
    
    E --> F[対策実施]
    F --> G[効果確認]
    G --> H{解消?}
    H -->|Yes| I[クローズ]
    H -->|No| E
```
