## 1. 業務概要

### 1.1 業務名称
第三者検証業務におけるドキュメントレビュー支援AIエージェントの設計開発に係る調査研究

### 1.2 業務目的
生成AIを活用したドキュメントレビュー支援AIエージェント（SmartReviewer）の技術的実現可能性を2ヶ月間の技術検証により明らかにすること。

### 1.3 業務期間
- 開始日: 2026年1月6日
- 終了日: 2026年3月5日（8週間）

### 1.4 期待成果
1. **技術的実現可能性の明確化**: 生成AIを活用したドキュメントレビュー支援の技術的実現可能性を、具体的な評価指標とともに明らかにする
2. **課題と対応方針の整理**: MVPレベルでのPoC実施を通じて、技術面・運用面・セキュリティ面等における具体的な課題を洗い出す
3. **年間計画試案の策定**: 庁内AI検証環境への統合、対象範囲の段階的拡大、機能の高度化に向けた計画を策定

## 4. 技術アプローチ

### 4.1 システムアーキテクチャ

Model Context Protocol (MCP) に基づくNeuro-Symbolicアーキテクチャを採用。

```
┌─────────────────────────────────────────────────────────────┐
│                    MCP Host (CLI/Web UI)                    │
└─────────────────────────┬───────────────────────────────────┘
                          │ MCP Protocol (JSON-RPC 2.0)
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ smartreviewer-  │ │ smartreviewer-  │ │ smartreviewer-  │
│ core            │ │ rag             │ │ knowledge       │
│                 │ │                 │ │                 │
│ Tools:          │ │ Tools:          │ │ Tools:          │
│ - review_doc    │ │ - vector_search │ │ - query_graph   │
│ - get_items     │ │ - graph_search  │ │ - check_ontol   │
│ - create_report │ │ - hybrid_retr   │ │ - get_coverage  │
└─────────────────┘ └─────────────────┘ └─────────────────┘
          │               │               │
          ▼               ▼               ▼
┌─────────────────────────────────────────────────────────────┐
│                      Data Layer                              │
│  Qdrant (Vector) │ Neo4j (Graph) │ MinIO │ RDF/OWL          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 主要技術スタック

| カテゴリ | 技術 | 選定理由 |
|----------|------|----------|
| プロトコル | MCP (Model Context Protocol) | AI統合の標準化、拡張性 |
| Vector DB | Qdrant | OSSで高性能、REST/gRPC対応 |
| Graph DB | Neo4j | 知識グラフに最適、Cypher対応 |
| オントロジー | RDFLib + Owlready2 | Python連携、OWL2対応 |
| LLM | 庁内AI環境（源内） | セキュリティ要件準拠 |

### 4.3 ハイブリッドRAGアプローチ

| チェック種別 | RAG方式 | 説明 |
|-------------|---------|------|
| 表記・フォーマット | Vector RAG | ガイドライン類似検索 |
| 構造・整合性 | Graph RAG | 知識グラフ探索 |
| 両方必要 | Hybrid RAG | Vector + Graph 組み合わせ |

## 6. 品質管理計画

### 6.1 品質目標

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| チェック合致率 | ≥ 70% | 評価データセットでの判定精度 |
| 処理時間 | ≤ 5分/件 | 10万文字文書での計測 |
| テストカバレッジ | ≥ 80% | pytest-cov |
| コードレビュー | 100% | PR必須レビュー |

### 6.2 品質保証活動

1. **テスト駆動開発 (TDD)**: MUSUBIX Article III準拠
2. **コードレビュー**: PRマージ前に必須
3. **CI/CD**: 自動テスト・静的解析
4. **回帰テスト**: 評価データセットでの継続的検証

### 8.2 連携方式

MCP Sampling機能を活用し、Host側でLLM呼び出しを実行。
これにより、MCP Server側にLLM認証情報を持たせる必要がなく、セキュリティ要件に適合。

```
┌─────────────────┐        ┌─────────────────┐
│   MCP Host      │        │  庁内AI環境     │
│                 │◄──────►│  （源内）       │

│  - 認証管理     │        │  - LLM API      │

         │ MCP Protocol
```
